# SA Weekly Digest - Manual Trigger

## Purpose

Run a weekly synthesis of the past week's `SA Daily Digest - <date>` pages from your Notion database. Reads the daily pages already in the database, surfaces themes that span multiple days, and writes a weekly summary page to the same database.

This skill is the "rollup" companion to `/sa-digest`. It reads structured properties from the daily database entries and reasons over their content to produce a cross-day synthesis - it does NOT re-scan GitHub, Notion keywords, partner conversations, or RSS feeds. The dailies have already done that work; this skill compounds them.

## Usage

- `/sa-weekly` - synthesize the **last full ISO week** (the most recent Monday-Sunday that has fully completed in UTC)
- `/sa-weekly 2026-05-07` - synthesize the ISO week **containing** this date (Mon-Sun)
- `/sa-weekly --dry-run` - run the full pipeline but write the markdown to a local file instead of creating a Notion page
- `/sa-weekly 2026-05-07 --dry-run` - both
- `/sa-weekly config` - show the current config (Notion IDs and sa-digest reuse)

The flags `--dry-run` and date arg can appear in any order. Dry-run output goes to `/tmp/team-digest-dry-runs/sa-weekly-<WEEK_LABEL>-v<N>.md`, ephemeral by design (compare once, discard).

This skill also runs from the terminal via `bin/sa-weekly-run.sh` in the team-digest repo. Same skill, same `--dry-run` flag.

## Important Runtime Notes

- **DO NOT use `readMcpResource` or `ReadMcpResourceTool`** to fetch Notion markdown specs. The output format is fully defined here.
- **Config is shared with sa-digest.** This skill reads the `sa-digest` key from `~/.config/team-digest/config.json` (Notion IDs, GitHub orgs, profile path). It does not require its own config block. The daily digests must have been generated by `/sa-digest` against the same `database_id`.
- **No network scanning.** No `gh search`, no Notion keyword search, no RSS fetches. The dailies have already done that. If you need fresh data, run `/sa-digest <date>` for the missing day first, then re-run `/sa-weekly`.

## Process

### Step 0: Argument parsing, subcommands, config load

Parse the skill argument as zero or more of:

- A `YYYY-MM-DD` date → captured as `$DATE_ARG`
- The literal `--dry-run` → set `$DRY_RUN=1`
- The literal `config` → handle as a subcommand (below)

Order does not matter: `/sa-weekly 2026-05-07 --dry-run` and `/sa-weekly --dry-run 2026-05-07` are equivalent.

#### Subcommand: `config`

Read `~/.config/team-digest/config.json`, display the `sa-digest` config (the same one this skill reads): Notion IDs (masked to last 8 chars), GitHub orgs, default keywords. Then stop.

#### Load config

Run the helper:

```bash
bash ~/.claude/skills/sa-digest/lib/load-config.sh sa-digest
```

Yes - this skill reads the **sa-digest** config, not a separate `sa-weekly` config. The two skills share Notion IDs, GitHub setup, and the team profile. This avoids config sprawl. The helper validates `notion.config_page_id` and `notion.database_id` are present and non-empty; on failure, surface the error and stop.

Also read the team profile at `~/.config/team-digest/profiles/sa-digest.md`. Used for the **SA Relevance** synthesis below. If absent, fall back to generic relevance heuristics.

### Step 1: Compute the week window

Run the helper:

```bash
eval "$(bash ~/.claude/skills/sa-weekly/lib/compute-week-window.sh "$DATE_ARG")"
# $WEEK_START, $WEEK_END, $WEEK_LABEL, $START, $END are now set.
```

`$WEEK_LABEL` is the ISO 8601 week label (e.g., `2026-W19`) used in the digest title. `$WEEK_START` and `$WEEK_END` are the Monday and Sunday dates of the week.

If the helper exits non-zero (invalid date format), surface its stderr to the user and stop.

### Step 2: Query the Notion database for daily digests in the window

Use the `notion-query-data-sources` MCP tool against the `data_source_id` of the SA Daily Digest database. The `data_source_id` is derived at runtime by fetching the database with `notion-fetch` (same pattern as sa-digest Step 0).

**Filter the query by:**

- `date:Date:start` between `$WEEK_START` and `$WEEK_END` inclusive
- `Digest Type` is `Combined` (the value `/sa-digest` writes for daily digests)

**Sort by `date:Date:start` ascending** so results come back in chronological order (Monday first).

**Result handling:**

- Each result has properties: `Digest Title`, `date:Date:start`, `Digest Type`, `Repos Active`, `Keywords Matched`, `Status`, `url`. These are CHEAP - use them as the primary signal for the "Week at a Glance" stats.
- Limit to 7 expected pages (one per weekday). If fewer found, that's the data - some days had no run. Note the gap in the digest body (e.g., "No daily digest for 2026-05-06 - run was missed or skipped.").
- If MORE than 7 found (multiple runs per day), use the LATEST one per `date:Date:start`. Older same-day pages are likely re-runs and contain stale data.

If the query returns zero results, abort with a clear message: `No daily digests found for week ${WEEK_LABEL} (${WEEK_START} to ${WEEK_END}). Generate the dailies first via /sa-digest <date>.`

### Step 3: Fetch each daily digest's content

For each unique daily page from Step 2, call `notion-fetch` with the page URL. **Parallelize.** Emit all calls in a single message so they run concurrently.

The response includes the page's full markdown content - the same content `/sa-digest` wrote. You will use this as the source of truth for synthesis.

**If any fetch fails:** log a one-line `(2026-05-06: not accessible - <reason>)` note and continue with the other days. The synthesis can still proceed with the remaining days.

### Step 4: Synthesize cross-day themes

This is the core work. For each theme below, read across all available daily digests and write a synthesis - NOT a copy-paste of the daily content. The point of the weekly is to surface what a reader could not get from any single daily.

**Theme A - Top GitHub themes**

Identify the 3-5 repos with sustained activity across the week. A repo with 1 PR on Monday is noise; a repo with PRs every weekday is signal. Write a 2-3 paragraph narrative for the top themes, linking to the most-significant individual PRs (the dailies already have the URLs). Surface architectural changes (the dailies will have called these out with Mermaid diagrams - reference the diagrams that ran during the week, do not re-render them). End the section with a one-line "Quieter this week" callout for repos that you'd normally expect activity from but didn't see (e.g., based on the team profile's "Our Key Repos" list).

**Theme B - Releases this week**

Collect every release mentioned in any daily digest's "Releases" section. Format as a single GFM table: `Repo | Version | Date | Notes`. Every entry must be linked: `[<tag>](release-url)` and `[<repo>](repo-url)`. Group rows by repo when a single repo had multiple releases.

**Theme C - Partner momentum**

Aggregate partner conversations across the week. For each company that appeared in 2+ daily digests, write a 1-2 sentence "what's moving" summary. For companies that appeared once, list them in a "Single touch" line at the end of the section. Pull action items the dailies surfaced and check if any spanned multiple days (e.g., a Tuesday action item that resurfaced as a Friday follow-up). Surface those as "Open threads" - they are the most likely SA escalation candidates.

**Theme D - Notion content pulse**

Aggregate the `Keywords Matched` property across all daily pages. Sort keywords by frequency (how many days they appeared). Top 3-5 keywords get a brief callout: "**EVM** appeared every day this week, mostly in [page-1](url) (Mon) and [page-2](url) (Wed)." Skip keywords that appeared on only one day - those are already well-covered in the daily.

**Theme E - Industry news roundup**

Aggregate the Industry News sections across all daily digests. **Deduplicate by URL** - a single blog post can appear in multiple dailies if its `pubDate` shifts due to timezone or feed lag. Group results by category (the same categories the dailies used: `hedera`, `ethereum`, `eips`). For each category, list 1 line per item with the link. If a category had nothing all week, omit the subsection.

**Theme F - Favorites movement**

Aggregate the "Favorites Activity" sections across all daily digests. **Pages that updated on 2+ days** get a callout - they are actively in flight. **Single-day updates** get a one-line bullet. Use the page URLs the dailies used so the reader can drill in. If "Pages I Created" sections fired across the week, list those too with the same multi-day-vs-single-day split.

**Theme G - Day-by-Day Index**

A simple list at the end: one line per daily digest, `- <weekday>, <date>: [<daily title>](<daily-page-url>)`. Lets the weekly serve as a navigation hub back into the dailies.

### Step 4.5: Pre-Write Link Audit (mandatory)

Before writing to Notion, scan the assembled digest content one final time. Verify:

1. **Every repo, PR, issue, and release reference is a markdown link.** Same rules as `/sa-digest` - bare entity references are unacceptable.
2. **Every Notion page reference is a markdown link.** All daily-digest links, all keyword-matched page links, all favorite-page links must use the URLs from the MCP responses.
3. **No `\n` inside Mermaid labels.** Same rule as `/sa-digest`. The weekly typically references diagrams from the dailies rather than rendering its own, but if it does render any (e.g., a "week at a glance" architecture flow), the rule applies.
4. **First-mention expansions are present.** When the synthesis names a project, component, or acronym for the first time, follow it with a 3-7 word plain-English expansion, sourced from the team profile's Project Glossary if present.
5. **Cross-day coherence.** If you mentioned a partner in Theme C and a related Notion page in Theme D, link the page from Theme C too. The weekly's value is the connections - make them clickable.

### Step 5: Write the weekly digest

**If `$DRY_RUN` is set:** do NOT call `notion-create-pages`. Instead, write the assembled markdown to a versioned local file:

```bash
DRY_DIR="/tmp/team-digest-dry-runs"
mkdir -p "$DRY_DIR"
N=1
while [ -f "$DRY_DIR/sa-weekly-${WEEK_LABEL}-v${N}.md" ]; do
  N=$((N + 1))
done
DRY_PATH="$DRY_DIR/sa-weekly-${WEEK_LABEL}-v${N}.md"
# Use the Write tool to write the assembled content to $DRY_PATH.
```

Write plain Markdown for the dry-run file (no Notion-flavored `<callout>`, `<details>`, `<table header-row>` tags). Convert callouts to blockquotes, sections to headings, GFM tables for tabular data. Echo the path to the user (`Dry-run output: <path>`) and stop.

**If `$DRY_RUN` is NOT set:** create a new page in the SA Daily Digest database using the `notion-create-pages` MCP tool.

**Parent:** `{ "type": "data_source_id", "data_source_id": "<data_source_id discovered in Step 2>" }`

**Properties:**
- `Digest Title`: `SA Weekly Digest - <WEEK_LABEL> (<WEEK_START> to <WEEK_END>)`
- `date:Date:start`: `<WEEK_END>` (Sunday - so weekly entries sort just after the last daily of that week)
- `Digest Type`: `Weekly`
- `Repos Active`: total unique repos active across all 7 dailies (sum-of-uniques, not sum-of-dailies)
- `Keywords Matched`: union of `Keywords Matched` across the week as a JSON array
- `Status`: `Auto`

**Content** uses Notion-flavored Markdown:

```
<callout icon="📈" color="purple_bg">
**SA Weekly Digest** | <WEEK_LABEL> | <WEEK_START> to <WEEK_END>
<N> daily digests processed | <N> repos active | <N> releases | <N> partner conversations
</callout>

---

# Week at a Glance

- N daily digests processed (Mon-Sun)
- N priority repos with sustained activity (3+ days)
- N releases shipped (linked below)
- N partners with conversations across the week
- N favorited pages with multi-day movement

---

# Top GitHub Themes

<2-3 paragraph synthesis from Theme A. Link every PR, issue, repo. End with one-line "Quieter this week" callout.>

---

# Releases This Week

<table header-row="true">
<tr><td>Repo</td><td>Version</td><td>Date</td><td>Notes</td></tr>
<tr><td>[<repo>](repo-url)</td><td>[<tag>](release-url)</td><td><date></td><td><short note></td></tr>
</table>

---

# Partner Momentum

<grouped by company. Multi-touch companies get 1-2 sentence "what's moving" summary; single-touch in a one-line "Single touch" footer. End with "Open threads" - action items spanning multiple days.>

---

# Notion Content Pulse

<top 3-5 keywords by frequency, with linked example pages>

---

# Industry News Roundup

<grouped by category, deduplicated. One line per item: - [<title>](<link>) - <source>>

---

# Favorites Movement

<multi-day pages first as callouts, single-day as bullets. Same dedup as Industry News.>

---

# Day-by-Day Index

- Monday, <date>: [<daily title>](daily-page-url)
- Tuesday, <date>: [<daily title>](daily-page-url)
- ...
- (gaps noted: "Wednesday, <date>: no daily digest run")

---

<callout icon="ℹ️" color="gray_bg">
**Auto-generated** by SA Weekly Digest | Synthesized from N daily digests | Week: <WEEK_LABEL> (<WEEK_START> to <WEEK_END>)
</callout>
```

## Style Rules

The same Style Rules from `/sa-digest` apply here. Specifically:

- **Synthesize, don't copy.** Re-summarizing the daily content is a waste; the point of the weekly is the cross-day signal you cannot get from any one daily.
- **Linking is mandatory.** Every entity reference (repo, PR, issue, release, Notion page, GitHub user) is a markdown link. See sa-digest's Linking Rules for the full table.
- **Plain-English first-mention expansion.** First mention of any project name, component, or acronym in a section gets a 3-7 word expansion. Use the team profile's Project Glossary as the source of truth.
- **No `\n` inside Mermaid labels.** If you render any Mermaid diagram (rare in weekly), keep all node labels on a single line.
- **Failure mode: partial digest, not no digest.** If one daily fetch fails, synthesize from the rest and note the gap. If one theme has no data, omit it from the output.

## Configuration

This skill reads from `~/.config/team-digest/config.json` under the `sa-digest` key (NOT a separate `sa-weekly` key). The shared config:

- `notion.database_id` - the same database the dailies were written to
- `notion.config_page_id` - read by `/sa-digest`, not used here directly
- `defaults.keywords` - used by `/sa-digest` for daily scanning, surfaced in this skill via the `Keywords Matched` property
- `github.orgs` - used by `/sa-digest`, surfaced in this skill via `Repos Active`

The team profile at `~/.config/team-digest/profiles/sa-digest.md` is shared too, including its **Project Glossary** for jargon expansion.

## Running headlessly (terminal / cron / launchd)

`bin/sa-weekly-run.sh` in the team-digest repo is the headless entry point. Same allow-listed Notion MCP tools as `bin/sa-digest-run.sh`, same env-var overrides for log paths and model.

```bash
bin/sa-weekly-run.sh                          # last full week
bin/sa-weekly-run.sh 2026-05-07               # week containing this date
bin/sa-weekly-run.sh --dry-run                # write to /tmp/team-digest-dry-runs/, skip Notion
bin/sa-weekly-run.sh 2026-05-07 --dry-run     # both
```

For automation: schedule `bin/sa-weekly-run.sh` on a Monday morning launchd/cron tick, AFTER the previous Friday's `/sa-digest` has had a chance to run. The skill expects all 5 weekday dailies to be in Notion already; if they're not, it produces a partial weekly with gap notes.
